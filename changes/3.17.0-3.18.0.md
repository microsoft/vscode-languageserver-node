# Features & API additions between `f1f8e1a` (3.17 merge, May 2022) and `HEAD` (main)

That range spans the 3.17.x line and the in‑progress 3.18 / 10.0.0 work for client and server. Below are only new features and public API additions — internal refactors and bug fixes are omitted.

## Protocol — new requests & notifications

- `textDocument/inlineCompletion` — inline completion request, params, registration options, and client capabilities. New file protocol.inlineCompletion.ts.
- `workspace/textDocumentContent` + `workspace/textDocumentContent/refresh` — server-provided dynamic text document content for custom schemes (multi-scheme registration supported). New file protocol.textDocumentContent.ts.
- `textDocument/rangesFormatting` — format multiple ranges in one request (`DocumentRangesFormattingRequest`, `DocumentRangesFormattingParams`).
- `workspace/foldingRange/refresh` — server‑initiated folding-range refresh (`FoldingRangeRefreshRequest`, `FoldingRangeWorkspaceClientCapabilities`).
- Notebook diagnostic pull and notebook code-action-kind support added on top of the existing notebook feature.

## Protocol - new features

### Workspace edits — `workspace/applyEdit`, `textDocument/rename`, code actions

- `WorkspaceEdit` may now carry `SnippetTextEdit` entries (snippet support in workspace edits). New client capability `workspace.workspaceEdit.snippetEditSupport`. See main.ts.
- `WorkspaceEditMetadata` added; `ApplyWorkspaceEditParams.metadata` carries it, with new client capability `workspace.workspaceEdit.metadataSupport`. Currently includes `isRefactoring`. See protocol.ts.
- `Command.tooltip` proposed on the existing `Command` type. main.ts.

### Completion — `textDocument/completion` / `completionItem/resolve`

- `CompletionList.itemDefaults` extended with `data` (per‑item default `data` payload).
- New `CompletionList.applyKind` (`CompletionItemApplyKinds`) describes per-field merge strategy (`Replace`/`Merge`) for `commitCharacters` and `data` between list defaults and individual items. Gated by new client capability `completionList.applyKindSupport`. See main.ts.
- `EditRangeWithInsertReplace` made a named type and exposed via `CompletionItemDefaults.editRange`.
- `CompletionItem.textEdit` may carry an `InsertReplaceEdit` (already in 3.17; capabilities cleaned up).
- All inline literal capabilities split into named types: `CompletionListCapabilities`, `CompletionClientCapabilities`, `ClientCompletionItemOptions`, `ClientCompletionItemOptionsKind`, `ClientCompletionItemResolveOptions`, `ClientCompletionItemInsertTextModeOptions`, `CompletionItemTagOptions`, `ServerCompletionItemOptions` (no behavior change but improved client/server symmetry).

### Code action — `textDocument/codeAction` / `codeAction/resolve`

- New `CodeAction.tags` field; `CodeActionTag.LLMGenerated` lets servers mark LLM‑generated actions. Client capability `textDocument.codeAction.tagSupport`. See protocol.ts.
- `CodeAction.disabled` reified as named `CodeActionDisabled` type.
- `CodeActionOptions.documentation: CodeActionKindDocumentation[]` — providers can ship docs that appear in the actions menu. Client capability `textDocument.codeAction.documentationSupport`.
- New code action kind `CodeActionKind.RefactorMove`.
- New code action kind `CodeActionKind.Notebook` (and notebook code‑action‑kind support in the client).
- Client capability literals replaced by named types (`ClientCodeActionKindOptions`, `ClientCodeActionLiteralOptions`, `ClientCodeActionResolveOptions`).

### Rename — `textDocument/rename` / `textDocument/prepareRename`

- `RenameParams` now extends `TextDocumentPositionParams` (instead of redefining textDocument/`position` inline).
- `PrepareRenameResult` reified as named types: `PrepareRenamePlaceholder` and `PrepareRenameDefaultBehavior` (server can return `{ defaultBehavior: true }` to ask the client to fall back to its default rename behavior).

### Formatting — `textDocument/rangeFormatting`

- `DocumentRangeFormattingClientCapabilities.rangesSupport` and matching `DocumentRangeFormattingOptions.rangesSupport`: a single capability flag that gates support for the new `textDocument/rangesFormatting` (multi‑range) request alongside the existing single‑range one.

### Diagnostics — `textDocument/publishDiagnostics` + pull (`textDocument/diagnostic`, `workspace/diagnostic`)

- `Diagnostic.message` may now be a `MarkupContent` (proposed). Client capability `textDocument.diagnostic.markupMessageSupport`. main.ts.
- `PublishDiagnosticsClientCapabilities` factored: shared bits hoisted into `DiagnosticsCapabilities` and reused by both push (`PublishDiagnosticsClientCapabilities`) and pull (`DiagnosticClientCapabilities`).
- `ClientDiagnosticsTagOptions` named type for `tagSupport`.

### Folding ranges — `textDocument/foldingRange`

- `FoldingRangeClientCapabilities` extended with named `ClientFoldingRangeKindOptions` and `ClientFoldingRangeOptions` (the latter exposes `collapsedText`).
- New workspace capability bucket `FoldingRangeWorkspaceClientCapabilities` (`refreshSupport`) — pairs with the new `workspace/foldingRange/refresh` request.

### Semantic tokens — `textDocument/semanticTokens/*`

- `SemanticTokenTypes.label` added.
- `SemanticTokenTypes` / `SemanticTokenModifiers` redefined as an open set (servers may report values outside the predefined list; clients must tolerate them).
- Inline capability literals replaced by named types: `ClientSemanticTokensRequestOptions`, `ClientSemanticTokensRequestFullDelta`, `SemanticTokensFullDelta`.

### Signature help — `textDocument/signatureHelp`

- Capability literals replaced by named types: `ClientSignatureInformationOptions`, `ClientSignatureParameterInformationOptions` (no functional change beyond improved client/server symmetry).

### Document & workspace symbols — `textDocument/documentSymbol`, `workspace/symbol`

- Capability literals replaced by named types: `ClientSymbolKindOptions`, `ClientSymbolTagOptions`, `ClientSymbolResolveOptions`.
- `WorkspaceSymbol.location` may now be `LocationUriOnly` (uri only, no range) — captured as a named type. main.ts.

### Code lens — `textDocument/codeLens` / `codeLens/resolve`

- `CodeLensClientCapabilities.resolveSupport: ClientCodeLensResolveOptions` — lets a client advertise which properties it can resolve lazily (mirrors the same shape on code actions/completion/symbols).

### Window / messages

- `MessageType.Debug` (5) added — used by `window/logMessage` (and accepted everywhere `MessageType` appears).
- `window/showMessageRequest` action items capability split into named `ClientShowMessageActionItemOptions`.

### Text document sync — `textDocument/didOpen` / `textDocument/didChange` / `textDocument/didClose`

- `TextDocumentContentChangeEvent` split into named variants: `TextDocumentContentChangePartial` (range-based) and `TextDocumentContentChangeWholeDocument` (full text). No wire change, but consumers now have stable type names.
- `didOpen` may be delayed and dropped if a matching `didClose` arrives in time (library-level optimization, surfaced via `TextDocumentClientCapabilities`).

### Document filters / file watching — affects every request that uses a `DocumentSelector`

- `DocumentFilter`/`NotebookDocumentFilter` reified into per-key variants (`…FilterLanguage` / `…FilterScheme` / `…FilterPattern`, and the notebook equivalents `NotebookDocumentFilterNotebookType` / `…Scheme` / `…Pattern`).
- `pattern` is now a `GlobPattern`, so registrations can use `RelativePattern`. Gated by `TextDocumentFilterClientCapabilities.relativePatternSupport`.
- `RegularExpressionEngineKind` introduced as an open kind type for `RegularExpressionsClientCapabilities.engine`.

### Markdown content — affects every request that returns `MarkupContent`

- `MarkdownClientCapabilities.supportThemeIcons` (`$(name)` icon syntax in markdown).
- Trusted‑markdown support extended: `Command` lists for `isTrusted` (`sanitizeIsTrusted` helper / `{ enabledCommands }` form) — applies wherever servers return markdown (hover, signature help, completion docs, etc.).

### Initialize — `initialize`

- `ClientInfo` and `ServerInfo` reified as named types (returned via `InitializeResult.serverInfo` / sent in `InitializeParams.clientInfo`).
- New named option types `StaleRequestSupportOptions` and `WorkspaceOptions` for the corresponding sections of `ServerCapabilities`.
- Capability info (server + client capability paths) attached to every existing request/notification namespace; `messageDirection` constant exposed on every existing message.

### Notebook document — `notebookDocument/didOpen` / `didChange` / `didSave` / `didClose`

- Granular notebook sync types added on top of the existing ones: `NotebookDocumentFilterWithNotebook`, `NotebookDocumentFilterWithCells`, `NotebookCellLanguage`, `NotebookDocumentCellChangeStructure`, `NotebookDocumentCellContentChanges`, `NotebookDocumentCellChanges`. These do not change the wire format but give the cell-change event a typed shape.
- Notebook documents now participate in diagnostic pull (`textDocument/diagnostic` and `workspace/diagnostic` may be issued for notebook cells).

### Document color — `textDocument/colorPresentation`

- Capability info added: clients now signal support via the standard capability-info path (`textDocument.colorProvider`), making client/server capability detection consistent with the other requests.

The corresponding library-level entries are in the “Next” section of README.md (snippet edits, workspace‑edit metadata, multiple range formatting, llmGenerated code actions, `MessageType.Debug`, `RefactorMove`, etc.).

## Protocol — new data types / shapes (mostly tagged `@since 3.18.0`)

- Workspace edits & rename
  - `SnippetTextEdit`, `WorkspaceEditMetadata`, `ChangeAnnotationsSupportOptions`.
  - `PrepareRenameResult` extended with `PrepareRenamePlaceholder` and `PrepareRenameDefaultBehavior`; `RenameParams` now references `TextDocumentPositionParams`.
- Code actions

  - `CodeActionKindDocumentation`, `CodeActionKind.RefactorMove` (proposed), `CodeActionDisabled`.
  - `ClientCodeActionKindOptions`, `ClientCodeActionLiteralOptions`, `ClientCodeActionResolveOptions`.
- Completion
  - `CompletionItemDefaults` (incl. `data`), `CompletionItemApplyKinds`, `ApplyKind` (`Replace` | `Merge`) — controls how list defaults merge with per‑item `commitCharacters`/`data`.
  - `EditRangeWithInsertReplace`, `CompletionItemTagOptions`, `CompletionListCapabilities`, `CompletionClientCapabilities`, `ClientCompletionItemResolveOptions`, `ClientCompletionItemInsertTextModeOptions`, `ClientCompletionItemOptions`, `ClientCompletionItemOptionsKind`, `ServerCompletionItemOptions`.
- Signature help: `ClientSignatureParameterInformationOptions`, `ClientSignatureInformationOptions`.
- Symbols / CodeLens: `ClientSymbolKindOptions`, `ClientSymbolTagOptions`, `ClientSymbolResolveOptions`, `ClientCodeLensResolveOptions`, `LocationUriOnly`.
- Semantic tokens: `SemanticTokenTypes.label`; `ClientSemanticTokensRequestFullDelta`, `ClientSemanticTokensRequestOptions`, `SemanticTokensFullDelta`; open-set semantics for `SemanticTokenTypes`/`SemanticTokenModifiers`.
- Markdown / messages: `MessageType.Debug`; `MarkdownString.supportThemeIcons`; trusted markdown command lists; `ClientShowMessageActionItemOptions`.
- Diagnostics: `DiagnosticsCapabilities`, `ClientDiagnosticsTagOptions`, refined `PublishDiagnosticsClientCapabilities`, `DiagnosticClientCapabilities`.
- Document/notebook filters
  - `TextDocumentFilterLanguage` / `…Scheme` / `…Pattern` (and the matching `NotebookDocumentFilter*` variants).
  - `GlobPattern` namespace + relative-pattern support; `TextDocumentFilterClientCapabilities`.
  - `NotebookDocumentFilterWithCells`, `NotebookDocumentFilterWithNotebook`, `NotebookCellLanguage`, finer-grained notebook change types.
- Inline completion data: `InlineCompletionItem`, `InlineCompletionList`, `InlineCompletionTriggerKind`, `InlineCompletionContext`, `SelectedCompletionInfo`, `StringValue`.
- Misc named types: `ServerInfo`, `ClientInfo`, `MessageType` literal, `RegularExpressionEngineKind` (namespaced kind type), `StaleRequestSupportOptions`, `WorkspaceOptions`, `LanguageKind` (open set; D, Pascal, Haskell added).
- Text content change variants: `TextDocumentContentChangePartial`, `TextDocumentContentChangeWholeDocument`.

## Meta model (`metaModel.json`) — schema-level features

- Per-request/notification capability info (server + client capability paths) — see messages.ts.
- `messageDirection` on all requests/notifications.
- `since` tags now support multiple values; `deprecated` field; `typeName` on notifications; option literals modeled as type definitions.
- Open-enum (`open`) marker for namespaced kind types.

## Client (`vscode-languageclient`) — new public API

- New middleware:
  - `TypeHierarchyMiddleware` (`PrepareTypeHierarchySignature`, `TypeHierarchySupertypesSignature`, `TypeHierarchySubtypesSignature`).
  - `InlineCompletionMiddleware`, `TextDocumentContentMiddleware`.
  - `GeneralMiddleware` — generic request/notification middleware.
  - `registerCapability` / `unregisterCapability` middleware hooks.
  - `applyEdit` in workspace middleware.
  - `resultReporter` substitution in `provideWorkspaceDiagnostics` middleware.
  - `formatMultipleRanges` on `DocumentRangeFormattingFeature`.
- `LanguageClient` surface:
  - `dispose()` for full client disposal; `ShutdownMode` enum; public start-failed state; auto-start on first `sendRequest`/`sendNotification`; handler registration (`onRequest`/`onNotification`/`onProgress`) before start.
  - Generic `ServerOptions`; `messageStrategy` option on connection.
  - Switch to `LogOutputChannel` for output/trace channel (log level read from the channel).
  - Added `CancellationToken` to the `showDocument` middleware (consistent shape across middleware).
  - `forgetDocument` on the diagnostic feature; peek-editor pull diagnostics.
- New client features: inline completion, dynamic text document content, snippet edits, notebook code action kinds.
- Browser client signature aligned with the Node client.

## Server (`vscode-languageserver`) — new public API

- `InlineCompletionFeature` (+ `InlineCompletionFeatureShape`).
- `TextDocumentContentFeature` (+ `TextDocumentContentFeatureShape`).
- `FoldingRangeFeature` exposed with an event emitter (required for the new refresh request).
- `NotebookDocuments<T>` class for managing notebook cell text documents.
- Server re-exports `vscode-languageserver-protocol` from `api.ts`.

## JSON-RPC (`vscode-jsonrpc`) — new public API

- `PortMessageReader` / `PortMessageWriter` — `MessagePort` transport.
- `MessageStrategy` connection option (`messageStrategy`) — pluggable message dispatch.
- `SharedArraySenderStrategy` / `SharedArrayReceiverStrategy` — `SharedArrayBuffer`‑based cancellation.
- `IdCancellationReceiverStrategy` / `RequestCancellationReceiverStrategy` (+ union `CancellationReceiverStrategy`).
- `TraceValue` namespace with new `Compact` level; `TraceValues` alias.
- New error codes: `PendingResponseRejected` (−32097), `ConnectionInactive` (−32096).
- `NoInfer`-based handler typings.
- New `RequestParam<P>`, `HandlerResult`, `NotificationResult` helper types.

## Engineering surface that affects API consumers

- Move to TypeScript 5.9, Node.js 22, ES2022; switch from `main`/`typings` to the `exports` field (`node16` module/moduleResolution) — affects tsconfig.json of consumers.
- `vscode-languageserver-types` ships separate UMD/ESM publishes; `module` export condition added so bundlers pick up the ESM build.
- `LinkedMap` no longer implements `Map` (breaking for code relying on the `Map` interface).
- Parallelism control for dispatch — notification handlers may now return a `Promise`.
- Notification handler registration can delay `didOpen` and skip it if `didClose` arrives in time.
